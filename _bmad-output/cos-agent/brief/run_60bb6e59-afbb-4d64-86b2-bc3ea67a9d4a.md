# Agent Studio — 제품 브리프

> CLI 기반 AI 에이전트 도구(OpenClaw 등)를 터미널 없이 브라우저에서 실행할 수 있는 웹 UI Wrapper 서비스

---

## 1. 문제 및 기회 (Problem / Opportunity)

OpenClaw와 같은 AI 에이전트 CLI 도구는 강력하지만, 터미널 명령어를 모르는 사용자에게는 진입 장벽이 높다. KiloClaw, StatClaw 같은 기존 Wrapper 서비스가 가능성을 보여줬지만, 프로젝트 기반 채팅 UX, 실시간 로그, 승인 확인 흐름을 통합한 제품은 없다. Agent Studio는 CLI 에이전트를 새로 만드는 것이 아니라, 기존 도구를 비개발자도 쓸 수 있는 웹 인터페이스로 감싸는 것에 집중한다.

---

## 2. 대상 사용자 및 사용 사례 (Target User / Use Case)

**대상 사용자:** CLI에 익숙하지 않지만 AI 에이전트 도구로 개발 작업을 자동화하고 싶은 운영자 및 비개발자 사용자

**주요 사용 사례:** 로컬 프로젝트 폴더를 등록하고, 채팅창에 자연어로 작업을 요청하면 CLI 에이전트가 실행되고, 결과·로그·파일 변경 내역을 브라우저에서 확인한다.

**핵심 흐름:** 프로젝트 선택 → 요청 입력 → 실행 결과 확인

---

## 3. 핵심 가치 제안 (Core Value Proposition)

**"터미널 없이, 채팅처럼 AI 에이전트를 사용한다."**

- CLI 명령어를 몰라도 브라우저에서 에이전트를 실행하고 결과를 확인할 수 있다.
- 실행 상태·로그·파일 변경·승인 요청이 하나의 채팅 흐름 안에서 명확하게 표시된다.
- 프로젝트별 실행 기록이 저장되어 이전 작업을 재확인하거나 재실행할 수 있다.

---

## 4. 성공 기준 정렬 (Success Criteria Alignment)

| 기능 영역 | "완료" 기준 |
|---|---|
| **프로젝트 등록 및 선택** | 웹 UI에서 로컬 폴더를 등록·목록 표시·선택할 수 있고, 프로젝트별 실행 기록이 분리 저장된다. |
| **CLI 실행 Wrapper** | 채팅 입력이 CLI 명령으로 변환·실행되며, 대기/실행/완료/실패 상태가 표시되고 실행 중 중단이 가능하다. |
| **실시간 로그 표시** | CLI 출력이 실시간으로 스트리밍되고, 파일 변경·명령 실행·오류가 구분 표시되며, 긴 로그는 접기/펼치기가 된다. |
| **채팅형 작업 요청** | 자연어 입력이 가능하고, 사용자-에이전트 대화 흐름이 저장되며, 이전 요청을 재실행 또는 수정 재실행할 수 있다. |
| **설정 관리** | CLI 경로·API 키·모델·실행 옵션을 설정·저장할 수 있고, 설정 테스트 버튼으로 CLI 실행 가능 여부를 확인한다. |
| **실행 기록** | 프로젝트별 기록 리스트가 저장되고, 상세 화면에서 로그·변경 파일·승인 내역을 확인하며 재실행이 가능하다. |
| **승인 확인** | 파일 수정·패키지 설치 등 민감 작업 시 채팅 내 승인 카드가 표시되고, 승인/거절 결과가 실행 기록에 남는다. |

---

## 5. 초기 범위 및 비목표 (Initial Scope and Non-Goals)

**In-scope**
- 위 7개 기능 영역 전체 (성공 기준 기반)
- 7개 화면: 온보딩, 프로젝트 선택, 메인 채팅/실행, 실행 로그 패널, 승인 UI, 실행 기록, 설정
- 로컬 실행 환경 (단일 사용자, 로컬 호스트 기반)

**Non-goals (명시적 제외)**
- 새로운 AI 에이전트 엔진 개발 — 기존 CLI 도구를 감싸는 것에만 집중
- 멀티 사용자 / 팀 협업 / 인증(Auth) 시스템
- 클라우드 배포 또는 원격 서버 실행
- 모바일 앱 (웹 브라우저 전용)
- CLI 도구 자체의 기능 확장 또는 수정

---

## 6. 핵심 아키텍처 제약 (Key Architecture Constraints)

아키텍처 결정은 이후 단계에서 내리지만, UX 브리프에서 다음 제약이 도출된다:

- **실시간 로그 스트리밍:** WebSocket 또는 SSE(Server-Sent Events)가 필요하다.
- **CLI 프로세스 관리:** 서버 사이드에서 CLI 프로세스를 실행·모니터링·중단할 수 있어야 한다.
- **로컬 파일 시스템 접근:** 프로젝트 폴더 경로를 서버가 읽을 수 있어야 하며, 브라우저 직접 접근이 아닌 서버 경유 방식이 필요하다.
- **설정 영속성:** API 키 등 민감 설정값의 로컬 저장소 보안을 고려해야 한다.
- **승인 흐름 동기화:** CLI 실행이 승인 응답을 기다리는 비동기 흐름이 필요하다.

---

## 7. 위험 및 미결 사항 (Risks / Decisions Needed)

1. **지원 CLI 도구 범위:** OpenClaw 외에 어떤 CLI 에이전트를 우선 지원할 것인가? 도구마다 출력 포맷이 달라 파싱 로직이 복잡해질 수 있다.

2. **브라우저 ↔ 로컬 파일 시스템 접근 방식:** 완전한 로컬 앱(Electron 등) vs 로컬 서버 + 브라우저 조합 중 어느 방향인지 결정이 필요하다.

3. **장시간 실행 프로세스 처리:** 에이전트가 수 분 이상 실행될 때 연결 유지, 타임아웃, 중간 상태 복원 전략이 필요하다.

4. **승인 UI의 인터랙션 신뢰성:** CLI 실행이 승인을 기다리는 동안 사용자가 페이지를 새로 고치거나 연결이 끊기면 어떻게 처리할 것인가?

5. **온보딩 최소 설정값 정의:** "필수 설정이 누락된 경우 안내한다"는 기준에서 어떤 설정이 절대 필수인지, 어떤 것이 선택인지 사전에 정의해야 한다.

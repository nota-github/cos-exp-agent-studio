**Depends On:** story-1.2, story-1.3

**In Scope:**

- `packages/client/src/hooks/useWebSocket.ts`: singleton WebSocket connection to `ws://localhost:3000/ws`; exponential backoff on disconnect (1s → 2s → 4s → 8s → 16s → max 30s); updates `wsConnected` and `reconnectAttempts` in Zustand store on connect/disconnect; exports `send(message)` and event emitter / event hook for incoming messages
- `packages/server/src/app.ts` updated to call WebSocket server setup after HTTP server creation

**Out of Scope:**

- The `connected` message with pending approvals (story-7.4)

**Acceptance Criteria:**
- Browser console shows "WebSocket connected" on page load
- Disconnecting the server causes client to log reconnect attempts with increasing delays
- Reconnecting the server causes client to reconnect and log "WebSocket connected" again
- `wsConnected` in Zustand store reflects true/false accurately
- Server logs client connect and disconnect events

**Validation Steps:**
1. Start both server and client with `npm run dev`

3. Stop server — verify client logs show reconnect attempts (check console)

### Project CRUD REST API

**Story Key:** story-2.1
**Epic Key:** epic-2
**Required for Success:** yes
**Depends On:** story-1.2

**Goal:** Expose REST endpoints for registering, listing, and deleting projects so the frontend can manage projects backed by persistent SQLite storage.

**In Scope:**
- `packages/server/src/routers/projectRouter.ts` mounted at `/api/projects`
- `GET /api/projects`: returns all projects ordered by `last_run_at DESC NULLS LAST`; each record includes `id, name, path, created_at, last_run_at, last_status`
- `POST /api/projects`: body `{name: string, path: string}`; zod validation (both required, name max 100 chars); verifies `path` exists on filesystem (`fs.existsSync`); generates UUID for `id`; inserts into projects table; returns 201 with created record
- `DELETE /api/projects/:id`: deletes project record; cascades: marks related executions/messages/logs/approvals as orphaned (or deletes if cascade configured); returns 204
- Path traversal prevention on the provided path: `path.resolve` normalization applied; must be absolute path
- 400 response with user-friendly message on validation failure; 404 on project not found; 409 if path already registered

**Out of Scope:**
- Project edit/rename (not in PRD scope)
- Filesystem browser for path selection (covered in story-2.2)
- Frontend UI (covered in stories 2.3 and 2.4)

**Acceptance Criteria:**
- `POST /api/projects` with valid name and existing path returns 201 with project record including generated `id`
- `POST /api/projects` with non-existent path returns 400 with message "경로가 존재하지 않습니다"
- `POST /api/projects` with already-registered path returns 409
- `GET /api/projects` returns array (empty array if none registered)
- `DELETE /api/projects/:id` removes the record; subsequent GET does not include it
- All inputs pass through zod validation; missing fields return 400 with field-level error messages

**Validation Steps:**
1. `curl -X POST localhost:3000/api/projects -H 'Content-Type: application/json' -d '{"name":"Test","path":"/tmp"}'` → 201 with id
2. `curl localhost:3000/api/projects` → array with one entry
3. `curl -X POST` with non-existent path → 400
4. `curl -X POST` same path again → 409
5. `curl -X DELETE localhost:3000/api/projects/{id}` → 204; GET returns empty array

**Affected Files or Modules:**
- `packages/server/src/routers/projectRouter.ts`
- `packages/server/src/app.ts` (mount router)
- `packages/server/src/db/index.ts` (query helpers or direct better-sqlite3 calls)

**Deferred Follow-ups:**
- Project edit (name change) is not in current PRD scope

### Filesystem directory explorer API

**Story Key:** story-2.2
**Epic Key:** epic-2
**Required for Success:** no
**Depends On:** story-1.2

**Goal:** Provide a server-side directory listing API so the frontend can offer a folder browser for project path selection without granting the browser direct filesystem access.

**In Scope:**
- `packages/server/src/routers/filesystemRouter.ts` mounted at `/api/filesystem`
- `GET /api/filesystem?path=<directory>`: lists immediate children of `path`; returns `{name: string, path: string, isDirectory: boolean}[]` with only directories included (files excluded)
- Security: `path.resolve(requestedPath)` must start with `os.homedir()` — requests outside home directory return 403
- Default path when query param omitted: `os.homedir()`
- Limit: returns max 200 entries (alphabetically sorted); does not recurse

**Out of Scope:**
- File content reading
- Paths outside home directory
- File listing (directories only for project selection use case)

**Acceptance Criteria:**
- `GET /api/filesystem` returns home directory contents (directories only)
- `GET /api/filesystem?path=/home/user/projects` returns subdirectories of that path
- `GET /api/filesystem?path=/etc` returns 403 (outside home)
- `GET /api/filesystem?path=../../etc` returns 403 (traversal attempt after resolve)
- Non-existent path returns 404

**Validation Steps:**
1. `curl "localhost:3000/api/filesystem"` → array of home subdirectories
2. `curl "localhost:3000/api/filesystem?path=/etc"` → 403
3. `curl "localhost:3000/api/filesystem?path=../../etc"` → 403 (resolved path check)
4. Verify only `isDirectory: true` entries are returned (no files)

**Affected Files or Modules:**
- `packages/server/src/routers/filesystemRouter.ts`
- `packages/server/src/app.ts` (mount router)

### ProcessManager — CLI process lifecycle management

**Story Key:** story-3.2
**Epic Key:** epic-3
**Required for Success:** yes
**Depends On:** story-3.1

**Goal:** Implement the ProcessManager that spawns, monitors, kills, and writes to CLI child processes, emitting events for log data and status changes that the rest of the system subscribes to.

**In Scope:**
- `packages/server/src/services/ProcessManager.ts`: class with:
  - `spawn(executionId: string, command: string, args: string[], cwd: string, env: NodeJS.ProcessEnv): void` — uses `child_process.spawn({shell: false, cwd, env})`; stores process reference keyed by executionId
  - `kill(executionId: string): void` — sends SIGTERM; after 5s if still running sends SIGKILL
  - `write(executionId: string, data: string): void` — writes to process stdin
  - `getStatus(executionId: string): ProcessStatus | null`
  - Emits events: `data(executionId, chunk, source: 'stdout'|'stderr')`, `exit(executionId, code, signal)`
- Server startup cleanup: on startup, ProcessManager queries DB for executions with `status='running'` or `status='pending'` and marks them `failed` with `error_message='서버 재시작으로 인해 중단됨'`
- ProcessManager is a singleton exported from the module

**Out of Scope:**
- Log writing to DB (covered in story-3.4)
- WebSocket broadcasting (covered in story-3.4)
- API endpoint wiring (covered in story-3.3)

**Acceptance Criteria:**
- `spawn` starts a real process (can be tested with `echo hello`) and `data` event fires with stdout content
- `kill` terminates the process; `exit` event fires; SIGKILL sent if process does not exit within 5 seconds of SIGTERM
- `write` sends data to process stdin (test with `cat` command reading stdin)
- Multiple concurrent processes managed by different executionIds without cross-contamination
- Server startup cleanup marks orphaned running executions as failed in DB

**Validation Steps:**
1. Unit test: spawn `echo "hello"` → data event received with "hello"
2. Unit test: spawn `sleep 60` → call kill → exit event fires within 6 seconds
3. Integration test: spawn execution, stop server, restart server → DB shows execution as failed with correct error message
4. Concurrent test: spawn two processes with different executionIds → each receives only its own data events

**Affected Files or Modules:**
- `packages/server/src/services/ProcessManager.ts`
- `packages/server/src/app.ts` (startup cleanup call)

### ApprovalService — signal detection and process pause

**Story Key:** story-7.1
**Epic Key:** epic-7
**Required for Success:** yes
**Depends On:** story-3.2, story-3.4

**Goal:** Detect when the CLI tool requests user approval, pause the CLI process, persist the approval request, and broadcast the event to the frontend.

**In Scope:**
- `packages/server/src/services/ApprovalService.ts`:
  - `parseApprovalSignal(line: string): ApprovalRequest | null` — tries JSON parse first; if parsed JSON has `{type: 'approval_request', action_type, target, risk_level, description}` shape, returns it; falls back to regex pattern matching against configurable whitelist patterns (e.g., `/Do you want to proceed\?/i`, `/Press y to confirm/i`); pattern-match creates a generic `ApprovalRequest` with `action_type='command_exec'`, `risk_level='medium'`
  - Integration with ProcessManager: on each `data` event (stdout line), pass through `parseApprovalSignal`; if not null: stop forwarding stdin to process (pause), insert `approvals` row (status=pending), insert `messages` row (type=approval_request, metadata={approvalId}), broadcast `{type:'approval_request', executionId, approval}` via WebSocket, update execution status=`approval_pending`
- Pattern list stored in `packages/server/src/services/approvalPatterns.ts` as exported constant array for easy extension

**Out of Scope:**
- REST endpoint for responding to approval (story-7.2)
- ApprovalCard UI (story-7.3)
- Reconnect restore (story-7.4)

**Acceptance Criteria:**
- When CLI outputs a line matching the JSON approval protocol, approval is detected
- When CLI outputs a line matching a whitelist pattern, approval is detected (fallback)
- Non-matching lines are NOT detected as approvals (no false positives for common log lines)
- Detected approval: DB approvals row inserted (status=pending), messages row inserted (type=approval_request), WS approval_request event broadcast, execution status=approval_pending
- CLI process receives no further stdin input while approval is pending

**Validation Steps:**
1. Inject a mock stdout line `'{"type":"approval_request","action_type":"file_modify","target":"/tmp/test.txt","risk_level":"high","description":"Overwrite test file"}'` into ProcessManager data handler → verify approval row in DB

4. Check WS client receives approval_request event with correct payload

**Affected Files or Modules:**
- `packages/server/src/services/ApprovalService.ts`
- `packages/server/src/services/approvalPatterns.ts`

**Story Key:** story-7.2
**Epic Key:** epic-7
**Required for Success:** yes

**Goal:** Expose an endpoint to submit the user's approval or rejection decision, triggering CLI continuation or cancellation.

  - `POST /api/approvals/:id/respond` body `{decision: 'approved'|'rejected'}` — zod validation

- `POST /api/approvals/:id/respond {decision:'rejected'}` → approval marked rejected, process killed
- Submitting to non-existent approval → 404
- Submitting to already-decided approval → 409
- After approval: `execution.status` back to 'running' in DB
- After rejection: `execution.status` is 'cancelled' in DB

4. POST to non-existent id → 404

## Implementation Summary

### Server: `executionRouter.ts`

**`redact(text)`** — replaces `/sk-ant-[A-Za-z0-9-]+/g` with `[REDACTED]`. Applied as the first operation on each chunk before any DB write or batch accumulation.

**`LogEntry` interface** — `{id, timestamp, level: 'info'|'warn'|'error', category: 'stdout'|'stderr'|'file_change'|'command', content}`.

**`logBatch`/`batchTimers`** — per-executionId Maps. On each data event, the entry is pushed to the batch and a 16ms debounce timer is armed (if not already running).

**`flushLogBatch(executionId)`** — broadcasts `{type:'log', executionId, data: LogEntry[]}` and cleans up both Maps. Called either by the debounce timer or eagerly at the top of the `exit` handler (to drain any unflushed entries before the completion broadcast).

**`exit` handler** — now begins with a timer cancel + eager flush before proceeding to DB update and completion broadcasts. This ensures log entries from the final output window are always delivered before `execution_complete`.

The old per-chunk `broadcast` call has been removed.

### Client: `logStore.ts` (new)

Zustand store keyed by `executionId`. Exports `LogEntry` type, `appendLogs`, and `clearLogs`. Used by `useWebSocket` for dispatch and will be consumed by LogPanel (story-4.2).

### Client: `useWebSocket.ts`

After dispatching to all registered `messageHandlers`, the singleton `onmessage` handler checks for `message.type === 'log'` and calls `useLogStore.getState().appendLogs(...)`. This keeps log buffering centralized in one place and available to any component that reads `useLogStore`.

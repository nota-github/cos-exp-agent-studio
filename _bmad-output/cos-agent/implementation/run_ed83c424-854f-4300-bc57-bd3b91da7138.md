## GUIDED_VERIFICATION_REQUIRED

Browser rendering of LogPanel toggle, accordion, and auto-scroll not confirmed at runtime.

Run `npm run dev` from the workspace root, then:

1. Navigate to `/projects/:id/chat` — confirm "로그 보기" button is visible in the top-right header area.
2. Click "로그 보기" — confirm LogPanel slides in from the right (animated, ~300ms), button label changes to "로그 닫기". The three sections (일반 로그 / 파일 변경 / 실행된 명령어) should be visible with empty-state text.
3. Click "로그 닫기" — confirm panel slides back off-screen.
4. Start an execution: `curl -X POST localhost:3000/api/executions -H 'Content-Type: application/json' -d '{"projectId":"<id>","requestText":"echo test"}'`
5. Subscribe via WS and observe logs arriving. With LogPanel open, confirm log entries appear in "일반 로그" section within ~100ms.
6. Produce a long output (>200 chars) — confirm entry shows truncated content with "펼치기" button; clicking shows full content; "접기" collapses it.
7. Produce a stderr/error — confirm red left border, red text, and ⚠ icon.
8. Scroll up inside the LogPanel while logs are streaming — confirm auto-scroll pauses. Scroll back to bottom — confirm auto-scroll resumes.

Report only if an unexpected error appears.

## Design Compliance Notes

**Status:** pass

- LogPanel does not occupy screen real estate when closed: `translate-x-full` positions it fully off-screen to the right.
- Panel open/close is animated via `transition-transform duration-300` — 300ms ease CSS transition.
- Error entries are visually distinguishable by red left border (`border-l-2 border-red-500`), red background tint (`bg-red-950/30`), red text (`text-red-300`), and `⚠` icon — not color-only differentiation.
- Three sections have clear visual separation via `border-b border-gray-800` dividers and sticky uppercase headers with distinct background.

## Design Differentiation Notes

**Status:** pass
**Intended archetype:** chat-first inbox

**Visible choices made:**
- LogPanel is a supplementary overlay that slides in only when requested — keeps the primary chat view uncluttered (aligns with "기본 화면을 복잡하게 만들지 않는다" UX principle)
- Toggle button uses ghost/outline style (`border border-gray-700`) to visually subordinate it to the Stop button (primary action)
- Section headers use monospace uppercase labels in muted gray — consistent with developer-tool aesthetic without overwhelming non-developer users
- Error log entries use multi-signal differentiation (border + background + text + icon) to make problems obvious at a glance

**Layout silhouette difference from default:**
- Right panel is a slide-over that doesn't shift page layout — the main chat area retains full width; the panel overlaps (z-20) rather than pushing content

**Default convergence risks avoided:**
- Did not create a persistent always-visible log sidebar that clutters the chat view
- Did not use a modal dialog for logs — kept them contextual and dismissible

**Remaining weak spots:**
- On very narrow viewports the 320px LogPanel may cover most of the chat area; no responsive breakpoint handling in this slice

---

## Outcome Quality Notes

**Purpose clarity:** The toggle button label ("로그 보기" / "로그 닫기") is specific and action-oriented; non-developer users understand what clicking it does.

**Primary action:** Log panel remains opt-in — the primary task (chat interaction) is unobstructed.

**State/copy handling:**
- Empty state: "로그 없음" / "변경 없음" / "명령 없음" per section — tells user what to expect
- Loading: no loading state needed; panel shows empty until data arrives via WebSocket

**Visible quality choices:**
- Sticky section headers let users see context as they scroll through long log lists
- `atBottomRef` ref-based auto-scroll avoids `useEffect` dependency array lint issues while preserving correct behavior

**Remaining weak spots:**
- Auto-scroll "resumed when scrolled back to bottom" relies on `handleScroll` updating `atBottomRef`; verified via code reasoning but not browser-confirmed

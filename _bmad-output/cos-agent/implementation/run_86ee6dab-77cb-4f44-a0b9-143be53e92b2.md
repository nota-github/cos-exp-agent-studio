## Quality Plan

| Criterion ID | Criterion | Intended Evidence |
|---|---|---|
| AC1 | `POST /respond {decision:'approved'}` when pending → approval marked approved, execution resumes | approvalRouter: `UPDATE approvals SET status='approved'`; `UPDATE executions SET status='running'`; `processManager.resume()` + `processManager.write('y\n')`; `broadcast({type:'status', status:'running'})` |
| AC2 | `POST /respond {decision:'rejected'}` → approval marked rejected, process killed | approvalRouter: `UPDATE approvals SET status='rejected'`; `UPDATE executions SET status='cancelled'`; `approvalService.removePending()`; `processManager.kill()` |
| AC3 | Non-existent approval → 404 | approvalRouter: DB lookup returns `undefined` → `res.status(404)` |
| AC4 | Already-decided approval → 409 | approvalRouter: `approval.status !== 'pending'` guard → `res.status(409)` |
| AC5 | After approval: `execution.status` = 'running' in DB | `UPDATE executions SET status = 'running'` before returning |
| AC6 | After rejection: `execution.status` = 'cancelled' in DB | `UPDATE executions SET status = 'cancelled'` before returning; exit handler also sets it to 'cancelled' on process kill signal |

## Outcome Quality Notes

- **Purpose clarity:** Endpoint is clearly scoped to the approval decision pathway; 404/409 give product-language Korean error messages consistent with other routers.
- **Primary action:** `POST /api/approvals/:id/respond` is the single action; zod validates the `decision` enum so invalid input returns 400 with field-level detail.
- **State/copy handling:** Korean error strings ("승인 요청을 찾을 수 없습니다", "이미 처리된 승인 요청입니다") match product copy conventions established in other routers.
- **Visible quality choices:** Resume-before-write ordering ensures the paused process flag is cleared before stdin write; `approvalService.removePending()` is called in both branches to keep the service state consistent.
- **Remaining weak spots:** GUIDED_VERIFICATION_REQUIRED for runtime curl tests.

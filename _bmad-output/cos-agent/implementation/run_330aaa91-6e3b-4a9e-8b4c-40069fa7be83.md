## Selected Story

**story-1.4 — WebSocket server and client singleton with reconnect**

Goal: Establish the bidirectional WebSocket channel between server and browser client, with automatic client-side reconnection, so all real-time features can build on this foundation.

---

## Quality Plan

Acceptance criteria mapped to code evidence (conservative mapping from story text; no machine-readable IDs extracted):

| Criterion | Code Evidence |
|-----------|--------------|
| Browser console shows "WebSocket connected" on page load | `useWebSocket.ts`: `ws.onopen` logs `[ws] WebSocket connected`; `useWebSocket()` hook calls `connect()` on mount |
| Disconnecting server causes client to log reconnect attempts with increasing delays | `scheduleReconnect()`: doubles `currentDelay` each attempt (1s→2s→4s→8s→16s→30s cap), logs attempt number |
| Reconnecting server causes client to reconnect and log "WebSocket connected" | `ws.onopen` resets `currentDelay` to 1s and logs the connected message |
| `wsConnected` in Zustand store reflects true/false accurately | `ws.onopen` calls `setWsConnected(true)`; `ws.onclose` calls `setWsConnected(false)` |
| Server logs client connect and disconnect events | `connection` handler logs `[ws] client connected`; `close` handler logs `[ws] client disconnected` |
| `broadcast(executionId, message)` and `broadcastAll(message)` exported | Both exported from `WebSocketServer.ts` |
| Server parses subscribe / unsubscribe messages | Message handler in `WebSocketServer.ts` handles both `subscribe` and `unsubscribe` message types |

## GUIDED_VERIFICATION_REQUIRED

Run `npm run dev` from the workspace root, then in a browser:

1. Open DevTools → Network → WS tab — verify WebSocket connection to `ws://localhost:3000/ws` appears
2. Console should show `[ws] WebSocket connected`
3. Stop the server process — console should show `[ws] reconnecting in 1000ms (attempt 1)`, then `2000ms`, `4000ms`, etc.
4. Restart the server — console should show `[ws] WebSocket connected` again within 30s

## In Scope for This Run

### Client — `packages/client/src/components/LogPanel.tsx`
- Added `import { useVirtualizer } from '@tanstack/react-virtual'` (already in package.json v3.11.2)
- Added `import type { LogEntry as LogEntryType } from '../stores/logStore'`
- Defined `const VIRTUAL_THRESHOLD = 200`
- Created `VirtualSection` component:
  - `useRef<HTMLDivElement>` as `parentRef` for virtualizer scroll element
  - `useVirtualizer({ count, getScrollElement, estimateSize: () => 44, overscan: 8 })`
  - `virtualizer.measureElement` ref on each item wrapper for dynamic height measurement
  - `virtualizer.scrollToIndex(entries.length - 1, { align: 'end' })` for auto-scroll in `useEffect` on `entries.length`
  - Scroll pause/resume via `atBottomRef` (same pattern as existing plain scroll)
  - Returns: `flex-1 overflow-y-auto min-h-0` container → `position: relative` inner div at `virtualizer.getTotalSize()` height → absolutely positioned virtual items with `translateY`
- Modified `LogPanel` component:
  - `const useVirtual = generalLogs.length > VIRTUAL_THRESHOLD`
  - **Virtual layout** (when `useVirtual`): general logs section is `flex-1 min-h-0` with `VirtualSection`; file_change and command sections are `flex-shrink-0` with `max-h-32 overflow-y-auto` plain renders
  - **Plain layout** (when `!useVirtual`): unchanged from story-4.2 — outer `overflow-y-auto`, three sections with sticky headers
  - `useEffect` for plain auto-scroll guarded with `!useVirtual` to avoid conflict

### Server — `packages/server/src/routers/executionRouter.ts`
- Added constants: `LOG_RETENTION_LIMIT = 10_000`, `LOG_TRIM_AMOUNT = 500`, `EXECUTION_RETENTION_LIMIT = 100`
- Added `logCounts = new Map<string, number>()` for in-memory per-execution log count tracking
- In `processManager.on('data')`: before DB INSERT, checks `currentCount >= LOG_RETENTION_LIMIT`; if true, deletes oldest 500 rows then inserts summary row `'[이전 10000줄 이상의 로그가 잘렸습니다]'` with level `'warn'`, category `'system'`; decrements count; then inserts new row and increments count
- In `processManager.on('exit')`: `logCounts.delete(executionId)` cleanup
- In `POST /executions` handler: after execution INSERT, counts project executions; if `execCount > EXECUTION_RETENTION_LIMIT`, selects oldest `execCount - 100` executions and manually cascade-deletes (logs → messages → approvals → executions)

## Deferred Work

- Browser fps confirmation at 5,000 entries requires manual DevTools testing (GUIDED_VERIFICATION_REQUIRED — see below)
- `logCounts` counter resets on server restart — existing in-progress executions will have `count = 0` until they restart. Acceptable for development tool context.
- The `'system'` category is new (not in client `LogEntry` type `category` union). Client renders it gracefully (falls into general logs section), but no category badge defined. Left as-is; a future story can add a badge for `'system'` category.

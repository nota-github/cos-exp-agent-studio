## In Scope for This Run

1. **Server** (`executionRouter.ts`): Added two new routes before `GET /:id`:
   - `GET /api/executions/:id/logs` — returns logs ordered by timestamp ASC
   - `GET /api/executions/:id/messages` — returns messages ordered by created_at ASC

2. **Client API** (`executions.ts`): Added types and hooks:
   - `Execution` interface
   - `useProjectExecutions(projectId)` — query for history list
   - `useExecution(executionId)` — query for detail view
   - `ExecutionLogEntry` interface
   - `useExecutionLogs(executionId)` — query for log accordion
   - `ExecutionMessage` interface
   - `useExecutionMessages(executionId)` — query for agent response

3. **ExecutionHistoryView.tsx** (was stub, now fully implemented):
   - Route: `/projects/:id/history`
   - Sidebar with `activeProjectId`
   - Loading / empty state
   - Execution rows: truncated request, date formatted as `YYYY.MM.DD HH:mm`, duration as `Xs` or `Xm Xs`, `ExecutionStatusBadge`
   - Failed rows: `border-l-4 border-l-red-500` red accent
   - Row click → navigate to `/executions/:id`

4. **ExecutionDetailView.tsx** (was stub, now fully implemented):
   - Route: `/executions/:id`
   - Sidebar with `activeProjectId={execution.project_id}`
   - Loading and not-found states
   - Sections: request, metadata, agent response, error, file changes, log accordion
   - Log accordion: collapsed by default, chevron rotation on expand, shows `LogEntry` components
   - File changes from `logs` where `category === 'file_change'`
   - Footer "다시 실행" button, disabled while submitting, navigates to chat on success

## Design Compliance Notes

**Status:** pass

- History list rows are scannable with status badge right-aligned, time and duration as secondary metadata below the request text — clear visual information hierarchy.
- Failed rows use a visible red left-border accent (`border-l-4 border-l-red-500`) matching the requested "실패한 실행 기록은 원인을 확인" design requirement.
- Empty state uses an icon + Korean copy ("아직 실행 기록이 없습니다") matching the copy strategy contract.
- Detail view has clear section hierarchy: request → response → logs → file changes, all collapsible to avoid cluttering the default view.

## Design Differentiation Notes

**Status:** pass  
**Intended archetype:** chat-first inbox  
**Visible choices made:**
- History view uses the same sidebar + header + content layout as ChatView — history feels like a contextual inbox panel, not a standalone admin screen.
- Row-as-message cards with hover state match the chat-inbox metaphor of scanning a message thread.
- Detail view places the request (user message analog) at top and response below, mirroring the chat bubble flow.

**Layout silhouette difference from default:** Left sidebar anchors the view; content area is max-w-3xl centered within the remaining space — same silhouette as ChatView, not a full-width table or card grid.

**Default convergence risks avoided:** Not a generic data table or CRUD list; rows are styled as message-like cards consistent with the inbox archetype.

**Remaining weak spots:** The detail view "다시 실행" button is in a footer strip — slightly more utilitarian than the chat-inline rerun buttons in story-6.4, but appropriate for a history read view.

## Outcome Quality Notes

- **Purpose clarity:** History view title "실행 기록" and empty-state copy immediately tell the user what the screen is and what to do.
- **Primary action:** "다시 실행" in detail view clearly named and consistently positioned.
- **State/copy handling:** Loading, empty, not-found states all covered with Korean product copy.
- **Visible quality choices:** Failed row red accent, log accordion collapse-by-default, file change purple pills, agent response styled differently from raw logs.
- **Remaining weak spots:** "다시 실행" failure (e.g. missing settings) is silently swallowed — but this is consistent with story scope; the settings gate is handled by story-6.3 chat flow.

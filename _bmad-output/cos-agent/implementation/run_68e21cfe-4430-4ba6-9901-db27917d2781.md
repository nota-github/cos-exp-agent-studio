**Epic:** epic-3 (CLI Execution Core)

---

## Quality Plan

No machine-readable criterion IDs were extracted. Conservative plan mapped from the story acceptance criteria:

| Criterion | Code Evidence |
|---|---|
| POST returns 201 + executionId within 200ms | executionRouter.ts: insert row, spawn fire-and-forget, return 201 — no await on CLI |
| POST with unregistered projectId returns 404 | DB lookup before insert; 404 if missing |
| POST when settings not configured returns 400 + missing fields | SettingsManager.isConfigured() called before spawn |
| DELETE kills running process; status → cancelled | processManager.kill(); 'exit' event handler updates status |
| GET /executions/:id returns correct record | direct DB lookup, 404 if not found |
| GET /projects/:id/executions returns scoped results | projectRouter.ts: SELECT WHERE project_id = ?, ORDER BY started_at DESC LIMIT 100 |

## Implementation Notes

**SettingsManager placement:** story-5.1 owns the full settings service; this run adds a minimal stub (JSON file only, no keytar) required for `isConfigured()`. story-5.1 can extend the class without breaking this slice.

**Route placement for GET /api/projects/:id/executions:** The story spec places this in executionRouter.ts, but Express cannot reach it there when projectRouter is mounted at `/api/projects` (the prefix intercepts the path first). The handler is placed in projectRouter.ts where it correctly receives `/:id/executions` after prefix stripping. Functionally identical to the spec.

**ProcessManager event wiring:** Registered at module level in executionRouter.ts (runs once on first import). Covers all executions spawned via the REST API.

**Design Compliance Notes:** not_applicable — story-3.3 is pure REST API, no UI changes.

**Design Differentiation Notes:** not_applicable — no UI.

## Design Differentiation Notes
- **Status: pass**
- **Intended archetype:** chat-first inbox
- **Visible choices made:** modal uses dark `gray-900` with `indigo-600` accents consistent with the inbox; "폴더 탐색" button visually toggles to indigo when active; directory browser uses folder SVG icons and chevron arrows; "선택 완료" CTA matches primary color
- **Layout silhouette difference from default/control:** modal is a dark rounded-xl overlay, not a white card form; directory browser is an inline accordion panel, not a separate dialog
- **Default convergence risks avoided:** no plain white form; no generic "Submit/Cancel"; field errors are inline; button copy is Korean and action-specific
- **Remaining weak spots:** directory browser path display shows `~` for home dir but shows full path when navigated — could confuse users unfamiliar with Unix paths

## Outcome Quality Notes
- **Purpose clarity:** AddProjectModal makes project registration visible and guided — no CLI knowledge required
- **Primary action:** "추가" button is prominent, disabled during pending with spinner, shows "추가 중..."
- **State/copy handling:** loading state ("추가 중..."), inline field errors, server 409 error propagated as Korean message from server body, successful submit closes modal and refreshes list
- **Visible quality choices:** `apiPostJson` captures server error body so Korean messages flow to UI without any client-side string mapping; `useCreateProject` automatically invalidates react-query cache on success; path-stack design in `DirectoryBrowser` avoids needing to know home dir from server
- **Remaining weak spots:** no optimistic update (list refreshes after server round-trip); directory browser cannot show why "위로 가기" is disabled at home level

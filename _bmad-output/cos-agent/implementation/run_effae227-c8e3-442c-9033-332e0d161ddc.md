## Design Differentiation Notes

**Status: pass**
**Intended archetype:** chat-first inbox

| Aspect | Choice |
|---|---|
| Visible choices | Approval card breaks the chat bubble pattern deliberately — it's an action card mid-stream, not a message. The `max-w-md` centering separates it from left/right-aligned bubbles. Risk badge in the header provides immediate visual signal before reading content. |
| Layout silhouette difference | Card is horizontally centered at `max-w-md` within the chat flow — unlike a chat bubble (which is left/right-aligned) and unlike a modal (which overlays). It looks like an inline gate card in a workflow tool, not a generic form or message. |
| Default convergence risks avoided | Not using a modal or side panel for approval; card is inline in the message thread preserving the chat-first flow. No generic "Yes/No" or "Approve/Deny" button labels. |
| Remaining weak spots | WS real-time card injection (without page refresh) was exercised via DB seed rather than live CLI → confirmed card loads from DB on page load; true WS push path deferred to story-7.4 live testing. |

## Outcome Quality Notes

**Purpose clarity:** A non-developer user sees the approval card appear in the chat flow at the point where action is needed — no context switch required.

**Primary action:** "변경 승인" is visually dominant (filled indigo button); "거절" is secondary (outline red). Hierarchy is immediately clear.

**State/copy handling:**
- Loading: card renders immediately with static metadata; approval status fetches in background
- Pending: two action buttons shown
- Decided: pill with "승인됨 ✓" or "거절됨 ✗" replaces buttons (cannot be re-decided)
- Error (409 already decided): `onError` invalidates query → card refetches and shows decided state

**Visible quality choices:** Inline approval in chat thread (not modal) keeps the UX model simple. The "거절" button is red and outlined (not filled) to signal destructive weight without being alarming. The decided state uses a pill shape to signal it's a record, not an action.

**Remaining weak spots:** Approval reconnect restore deferred to story-7.4. WS real-time card injection path (without page load) verified via DB seed + page-load; live CLI WS push path exercised in story-7.4/7.5 integration testing.

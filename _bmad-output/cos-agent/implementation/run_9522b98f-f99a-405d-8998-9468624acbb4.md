## Selected Story

**story-3.4:** Execution state persistence and WebSocket status events

Goal: Persist execution state transitions to the database as CLI process events arrive, and broadcast status changes via WebSocket so the frontend receives real-time execution state updates.

---

## Quality Plan

Story-3.4 had no machine-readable criterion IDs. Conservative plan mapped from the story acceptance criteria:

| Criterion | Intended Evidence |
|---|---|
| DB row transitions pending → running → completed/failed/cancelled | executionRouter.ts: status set to 'pending' at INSERT, 'running' before spawn, then updated by exit handler |
| `projects.last_run_at` and `last_status` updated after completion | exit handler: `UPDATE projects SET last_run_at = ?, last_status = ? WHERE id = ?` |
| WebSocket clients receive `{type:'status'}` messages for each transition | broadcast called with `{type:'status', executionId, status:'running'}` after spawn; broadcast `{type:'status', executionId, status}` in exit handler |
| Exit code 0 → completed; non-zero → failed; signal → cancelled in DB | `const status = signal ? 'cancelled' : code === 0 ? 'completed' : 'failed'` |
| `error_message` set to last stderr chunk on failed exit | `lastStderrChunk` map tracks last stderr per execution; written to DB when `status === 'failed'` |

---

## In Scope for This Run

Story-3.3 had already implemented most of story-3.4's DB persistence requirements (log writes, exit status update, project last_run_at). The delta for story-3.4 was:

**1. `{type:'status'}` WebSocket status broadcasts (missing in 3.3):**
- After successful `processManager.spawn()`: `broadcast(executionId, {type:'status', executionId, status:'running'})`
- In `exit` event handler: `broadcast(executionId, {type:'status', executionId, status})` before `execution_complete`

**2. `error_message` set to last stderr chunk on failure (missing in 3.3):**
- Added `const lastStderrChunk = new Map<string, string>()` accumulator
- In `data` handler: `lastStderrChunk.set(executionId, chunk)` when `source === 'stderr'`
- In `exit` handler: `lastStderrChunk.get(executionId)` passed as `error_message` when `status === 'failed'`
- DB UPDATE extended: `SET status = ?, completed_at = ?, duration_ms = ?, summary = ?, error_message = ?`

**Modified files:**
- `packages/server/src/routers/executionRouter.ts` — added `lastStderrChunk` map, stderr tracking in data handler, `error_message` in exit UPDATE, two `broadcast({type:'status',...})` calls

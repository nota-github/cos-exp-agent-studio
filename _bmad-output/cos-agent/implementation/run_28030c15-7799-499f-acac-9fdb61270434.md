## Selected Story

**story-6.4** — Re-run and edit-and-rerun from chat messages

Goal: Let users repeat a previous task or modify it before re-running, so they can iterate on failed or incomplete work without retyping.

## In Scope for This Run

### MessageBubble.tsx
- Added `onRerun?: () => void`, `onEditRerun?: (text: string) => void`, `canRerun?: boolean` to Props interface
- For `type='user'`: outer div gains `group` class and `relative` positioning
- Absolutely-positioned action container (`left-4 top-1/2 -translate-y-1/2`) with `opacity-0 group-hover:opacity-100 transition-opacity duration-150`
- "수정 후 실행" button: calls `onEditRerun?.(content)`; "다시 실행" button: calls `onRerun?.()`
- Both buttons: `disabled={!canRerun}` with muted resting style, clear hover style
- Action container only rendered when `onRerun != null || onEditRerun != null`

### InputBar.tsx
- Added `prefillText?: string` and `prefillKey?: number` to Props interface
- Added `useEffect` watching `[prefillKey, prefillText, resize]`: when `prefillKey` is truthy, directly sets `textareaRef.current.value`, calls `resize()`, calls `textareaRef.current.focus()`, then updates React state via `setValue()`

### ChatView.tsx
- Added `editState` state: `{ text: string; key: number } | null`
- Added `handleEditRerun` callback: `setEditState(prev => ({ text, key: (prev?.key ?? 0) + 1 }))`
- Added `canRerun = !canStop && !submitting` derived boolean
- MessageBubble render: passes `onRerun`, `onEditRerun`, `canRerun` only for `type='user'` messages
- InputBar: passes `prefillText={editState?.text}` and `prefillKey={editState?.key}`

**Status: pass**

- Layout silhouette: buttons appear within the chat row without changing the layout — consistent with inbox message interaction patterns
- Default convergence risks avoided: no generic "Retry" label; actions are inline in the chat, not a side panel or modal

## Outcome Quality Notes

- **Purpose clarity:** Non-developer users can iterate on tasks without retyping — the "다시 실행" / "수정 후 실행" affordance is discoverable via hover
- **Primary action:** Re-run is wired to the same `handleSubmit` path — consistent with the existing submit flow
- **State/copy handling:** Buttons are disabled during execution (canRerun guard) — users cannot accidentally trigger a second run while one is active
- **Visible quality choices:** Absolute positioning for no layout shift; opacity fade for smooth appearance
- **Remaining weak spots:** GUIDED_VERIFICATION_REQUIRED items; focus behavior on mobile is untested

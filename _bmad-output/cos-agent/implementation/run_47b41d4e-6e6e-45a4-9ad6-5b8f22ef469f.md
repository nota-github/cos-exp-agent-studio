### GUIDED_VERIFICATION_REQUIRED — browser rendering not confirmed at runtime

Run `npm run dev` from the workspace root, then perform these steps:

1. Navigate to `/projects/:id/chat` where `:id` is a registered project ID — confirm header shows project name; no status badge (no execution active).
2. Start an execution via API: `curl -X POST localhost:3000/api/executions -H 'Content-Type: application/json' -d '{"projectId":"<id>","requestText":"echo hello"}'` — note the returned `executionId`.
3. Navigate to `/projects/:id/chat?executionId=<executionId>` — confirm the blue "실행 중" badge with spinner appears; confirm red "실행 중단" button is visible.
4. Click "실행 중단" — confirm badge transitions to "취소됨" within 2 seconds; Stop button disappears.
5. Start another execution and let it finish — confirm badge changes to "완료"; Stop button not visible.
6. Navigate without `?executionId` — confirm no badge is shown (status is null).

No code changes are expected. Report only if an unexpected error appears during these steps.

## Outcome Quality Notes

- **Purpose clarity:** Header now communicates current project and live execution state at a glance — a first-time user sees "project name + 실행 중 (spinner)" and understands something is running.
- **Primary action:** Stop button is red and labeled "실행 중단" (not generic "Stop") — specific to the user situation.
- **State/copy handling:** Status null → no badge (neutral). Running → blue badge + spinner + red Stop button. Terminal states → colored badge, no Stop. Copy is Korean and action-specific throughout.
- **Visible quality choices:** Spinner animation on running state; disabled + opacity-50 for the Stop button during 1s cooldown; per-state Tailwind color classes with good contrast.
- **Remaining weak spots:** Chat area shows a placeholder paragraph — content will arrive with story-6.2. Status can only be observed if a real execution is running and URL param is provided.

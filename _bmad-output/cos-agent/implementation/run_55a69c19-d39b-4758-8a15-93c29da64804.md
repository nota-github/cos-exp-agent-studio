## Implementation Details

### `packages/client/src/api/settings.ts` (new)

Thin API module over the existing `apiGet`/`apiPut`/`apiPost` client:

```typescript
getSettings()   // GET /api/settings → SettingsConfig (includes apiKeySet: boolean)
saveSettings()  // PUT /api/settings → SettingsConfig
testCli()       // POST /api/settings/test → TestResult
```

### `packages/client/src/views/SettingsView.tsx` (replaced)

**Layout:** Matches the chat-first inbox pattern established by `ProjectInboxView`:
- Narrow left icon rail (w-14, indigo logo, nav icons) — projects icon + settings icon (active/highlighted)
- Fixed header bar with title "설정" and subtitle
- Scrollable main content area (`max-w-2xl mx-auto`)

**Form sections (6):**
1. **CLI 도구 실행 경로** — monospace text input + "찾아보기" button opening inline `DirectoryBrowser`
2. **API 키** — `type="password"` input; "저장됨 ✓" badge (emerald, check icon) visible when `apiKeySet && !form.apiKey`
3. **기본 모델** — `<select>` with 4 presets (Opus 4.7, Sonnet 4.6, Haiku 4.5, 직접 입력) + conditional custom text input
4. **기본 프로젝트 폴더** — monospace text input + "찾아보기" button with own `DirectoryBrowser`
5. **기본 실행 옵션** — monospace text input for extra CLI flags
6. **실행 기록 보존** — `type="number"` input (min=1, max=500)

**Action bar:**
- "설정 테스트" — calls `testCli()`, spinner during test, green inline result panel on success ("CLI 도구가 정상 작동합니다" + version output), red inline panel on failure with server error message
- "저장" — calls `saveSettings()`, spinner during save, inline confirmation or error message (auto-dismisses after 4s on success)
- Load error (if `GET /api/settings` fails) shown as red banner at top of form

**State management:** All local `useState` — no global store changes needed. API key clears after save to avoid displaying masked value back.

## Design Differentiation Notes

**Status: pass**

- **Intended archetype:** chat-first inbox
- **Visible choices made:**
  - Narrow icon rail (w-14) with logo + nav icons — same silhouette as ProjectInboxView and ChatView, maintaining the inbox chrome across all views
  - Header bar with title + subtitle follows the same pattern as the rest of the app
  - No generic "Settings" page chrome; inherits the dark sidebar-first shell that unifies the product
  - Settings icon (gear) highlighted with indigo color + `bg-gray-700/60` background to signal active state
- **Layout silhouette difference from default:** Sidebar rail + fixed header + scrollable content area — clearly a panel in the inbox shell, not a standalone centered form
- **Default convergence risks avoided:** No generic centered white card, no top-navigation tabs, no plain default `<form>` page
- **Remaining weak spots:** The settings form itself is necessarily form-like; the differentiation is primarily in the outer shell and icon rail consistency

## Outcome Quality Notes

- **Purpose clarity:** Header "설정" with subtitle "CLI 도구 및 API 설정을 관리합니다" immediately explains the page
- **Primary action:** "저장" is the primary CTA; indigo color distinguishes it from secondary "설정 테스트" (gray)
- **State/copy handling:**
  - Empty state: placeholder copy in each input explains expected format (e.g., `/usr/local/bin/claude`)
  - Loading: `GET /api/settings` failure shows "설정을 불러오는 중 오류가 발생했습니다" as a red banner
  - Error: Save errors shown inline below "저장" button; test errors shown inline below "설정 테스트" button
  - Success: "설정이 저장되었습니다" auto-dismisses after 4 seconds; "CLI 도구가 정상 작동합니다" stays until next test
- **Visible quality choices:** Password masking, "저장됨 ✓" indicator, auto-dismiss on save success, spinner during async ops
- **Remaining weak spots:** DirectoryBrowser for CLI path selects a directory; user still needs to manually append the binary name (by design, per story spec)

## Selected Planning Contract

### rigorous_planning (꼼꼼하게 만들기)

**Rationale:**

Agent Studio는 CLI 서브프로세스 실행, API 키 저장, 승인 워크플로우, 실시간 스트리밍, 복잡한 다중 상태 흐름이 결합된 프로젝트로, 다음 escalation 신호들이 rigorous_planning을 정당화한다.

1. **External CLI Integration (weight=3):** 사용자 채팅 입력이 CLI 명령어로 변환되어 서버에서 서브프로세스로 실행된다. 사용자가 제공한 경로와 입력값이 CLI 인수로 사용되므로 command injection 위험이 있다. 입력 검증과 실행 경계 설계가 필수적으로 필요하다.

2. **Sensitive Data — API Key Storage (weight=3):** 설정 화면에서 API 키 및 인증 정보를 저장한다. 로컬 환경이더라도 민감 데이터 저장 방식(암호화, 로그 노출 방지 등)에 대한 명시적 설계가 필요하다.

3. **Permissioned Domain — 승인 워크플로우:** 파일 수정, 패키지 설치, 외부 명령 실행에 대한 사용자 승인 게이트가 있다. 승인 대기 중 연결 끊김, 페이지 새로고침, 거절 후 상태 복원 등의 edge case가 존재한다.

4. **Complex Multi-State Workflow:** 7개 화면, pending → running → complete/failed 상태 전이, 중단/재실행 기능, WebSocket/SSE 기반 실시간 로그 스트리밍, 프로젝트별 실행 기록 분리 저장.

5. **Architecture Risks:** 브리프 섹션 7(위험 및 미결 사항)에서 CLI 도구 범위, 장시간 실행 처리, 승인 UI 신뢰성, 온보딩 최소 설정값 정의 등 4가지 미결 결정사항이 식별됨. 이는 rigorous 단계의 edge case 및 risk 문서화가 필요함을 시사한다.

6. **Process Depth Recommendation:** 코드베이스 신호 점수 — external_integration(3) + sensitive_or_permissioned_domain(3) + ambiguous_or_broad_scope(2) + data_persistence(2) + multi_state_flow(2) = 12점, rigorous_build 신뢰도 High.

7. **Build Mode Preference:** codee_recommended (코디 추천) — 가장 안전하고 적합한 계약 선택. CLI 실행 표면과 API 키 처리를 고려할 때 rigorous_planning이 가장 안전한 선택이다.

**Why not standard_planning?**
standard_planning은 민감/권한 도메인이 없고 리스크가 낮을 때 적용된다. CLI 서브프로세스 실행(사용자 입력 → CLI 인수)과 API 키 저장의 조합은 standard_planning의 risk threshold를 초과한다.

**Why not plan_lite?**
plan_lite는 1-3개 화면, 1-3개 핵심 액션, 외부 통합 없음이 조건이다. Agent Studio는 7개 화면, 외부 CLI 통합, API 키 저장을 포함하므로 해당 없다.

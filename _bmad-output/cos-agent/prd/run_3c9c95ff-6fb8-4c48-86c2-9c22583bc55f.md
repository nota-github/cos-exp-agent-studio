## 0. Document Purpose

This PRD translates the approved Agent Studio brief and project success criteria into product-level requirements. It is specific enough for an architect to derive technical constraints, but does not prescribe implementation architecture or create implementation stories — those belong in later phases.

---

## 1. Target User and Use Case

**Target User:** CLI 환경에 익숙하지 않은 운영자 및 비개발자 사용자로, AI 에이전트 도구(OpenClaw 등)로 로컬 프로젝트의 개발 작업을 자동화하고 싶지만 터미널 명령어를 직접 다루기 어려운 사람.

**Primary Use Case:** 로컬 프로젝트 폴더를 브라우저에서 등록하고, 채팅창에 자연어로 작업 요청을 입력하면 CLI 에이전트가 실행되어 결과·로그·파일 변경 내역을 브라우저에서 실시간으로 확인한다.

**Core Task Flow:** 프로젝트 선택 → 요청 입력 → 실행 결과 확인

**Completion Moment:** 에이전트가 작업을 완료하고 최종 응답과 변경 내역이 채팅 화면에 표시되면 사용자는 결과를 확인하고 다음 요청을 입력하거나 실행 기록을 저장한다.

## 2. Goals

| # | Goal |
|---|------|
| G1 | CLI를 모르는 사용자가 터미널 없이 브라우저만으로 AI 에이전트를 실행할 수 있게 한다. |
| G2 | 프로젝트 기반 채팅 UX를 통해 "프로젝트 선택 → 요청 입력 → 실행 결과 확인" 흐름을 단순하게 유지한다. |
| G3 | CLI 실행 상태(대기 중/실행 중/완료/실패)와 실시간 로그를 채팅 흐름 안에서 직관적으로 표시한다. |
| G4 | 민감한 작업(파일 수정, 패키지 설치 등)은 사용자 승인 확인 절차를 거쳐 안전하게 실행한다. |
| G5 | 프로젝트별 실행 기록을 저장하여 이전 작업 확인 및 재실행이 가능하게 한다. |
| G6 | 첫 실행 온보딩으로 비개발자 사용자가 필수 설정을 완료하고 바로 사용할 수 있게 안내한다. |

---

## 3. Non-Goals

| # | Non-Goal | Reason |
|---|----------|--------|
| NG1 | 새로운 AI 에이전트 엔진 개발 | 기존 CLI 도구 Wrapper에만 집중 |
| NG2 | 멀티 사용자 / 팀 협업 / 인증(Auth) 시스템 | 단일 사용자 로컬 실행 환경 범위로 제한 |
| NG3 | 클라우드 배포 또는 원격 서버 실행 | 로컬 호스트 기반 환경으로 한정 |
| NG4 | 모바일 앱 | 웹 브라우저 전용 |
| NG5 | CLI 도구 자체의 기능 확장 또는 수정 | CLI 도구는 블랙박스로 취급 |
| NG6 | 완전한 IDE 기능(코드 편집, diff 뷰어, 브랜치 관리 등) | 실행 결과 확인과 로그 표시까지만 담당 |

## 4. Core Requirements

### 4.1 프로젝트 등록 및 선택 (Feature Area: Project Management)

**FR-PM-1:** 사용자가 웹 UI에서 로컬 프로젝트 폴더 경로를 입력하거나 탐색하여 등록할 수 있어야 한다.  
**FR-PM-2:** 등록된 프로젝트 목록이 카드 또는 리스트 형태로 표시되며, 각 항목에는 프로젝트 이름, 로컬 경로, 최근 실행 시간, 최근 상태가 표시된다.  
**FR-PM-3:** "새 프로젝트 추가" 버튼을 통해 프로젝트를 등록할 수 있어야 한다.  
**FR-PM-4:** 프로젝트를 선택하면 해당 프로젝트를 기준으로 CLI 에이전트가 실행되는 메인 채팅/실행 화면으로 이동한다.  
**FR-PM-5:** 프로젝트별 실행 기록이 분리되어 저장된다.  
**FR-PM-6 (Empty State):** 등록된 프로젝트가 없을 때 사용자에게 첫 번째 프로젝트를 등록하도록 안내하는 메시지를 표시한다.

---

### 4.2 CLI 실행 Wrapper (Feature Area: Execution Engine)

**FR-EE-1:** 사용자가 채팅창에 입력한 자연어 요청이 선택된 프로젝트 경로 및 설정값과 결합되어 실제 CLI 명령으로 변환·실행된다.  
**FR-EE-2:** CLI 실행 상태가 **대기 중 / 실행 중 / 완료 / 실패** 4가지 상태로 표시된다.  
**FR-EE-3:** 실행 중인 작업을 사용자가 "중단" 버튼으로 종료할 수 있다. 실행 중이 아닐 때는 중단 버튼이 숨겨진다.  
**FR-EE-4:** CLI 실행 실패 시 에러 메시지와 관련 로그가 채팅 화면에 표시된다.  
**FR-EE-5 (Security Constraint):** CLI 명령 구성 시 사용자 입력을 직접 쉘에 연결하지 않는다. 명령 인수는 반드시 파라미터화되어야 한다 (Command Injection 방지).

---

### 4.3 실시간 로그 및 결과 표시 (Feature Area: Live Output)

**FR-LO-1:** CLI 실행 중 출력되는 로그가 웹 UI에 실시간으로 스트리밍된다.  
**FR-LO-2:** 에이전트의 최종 응답이 채팅 메시지 형태로 표시된다.  
**FR-LO-3:** 실행 결과 내 **파일 변경 / 명령 실행 / 오류** 항목이 시각적으로 구분된다.  
**FR-LO-4:** 긴 로그는 기본 요약 표시로 시작하며, 사용자가 펼치면 전체 내용을 볼 수 있다 (접기/펼치기).  
**FR-LO-5:** 오류 로그는 일반 로그와 구분되는 스타일(색상, 아이콘 등)로 표시된다.  
**FR-LO-6:** 실행 로그 패널은 우측에 위치하며 필요할 때만 열린다 (기본 화면을 복잡하게 만들지 않음).

---

### 4.4 채팅형 작업 요청 (Feature Area: Chat Interface)

**FR-CI-1:** 사용자가 메신저처럼 자연어로 작업 요청을 입력할 수 있어야 한다.  
**FR-CI-2:** 사용자의 요청과 에이전트 응답이 대화 흐름(타임라인)으로 표시되고 저장된다.  
**FR-CI-3:** 동일 프로젝트 안에서 이전 대화 기록을 스크롤하여 다시 확인할 수 있다.  
**FR-CI-4:** 이전 요청을 원클릭으로 재실행하거나 수정하여 재실행할 수 있다.  
**FR-CI-5 (UX Constraint):** 채팅 입력창과 실행/중단 버튼이 항상 하단에 고정 표시된다.

---

### 4.5 설정 관리 (Feature Area: Configuration)

**FR-CF-1:** CLI 도구 실행 경로를 설정할 수 있어야 한다.  
**FR-CF-2:** API 키, 기본 모델, 기본 프로젝트 폴더, 기본 실행 옵션을 설정할 수 있어야 한다.  
**FR-CF-3:** 설정값은 영속적으로 저장되어 앱을 다시 열어도 유지된다.  
**FR-CF-4:** 필수 설정이 누락된 경우 실행 전에 어떤 설정이 필요한지 사용자에게 명확히 안내한다.  
**FR-CF-5:** "설정 테스트" 버튼으로 현재 설정으로 CLI 도구가 정상 실행되는지 확인할 수 있다.  
**FR-CF-6 (Security Constraint):** API 키 등 민감한 설정값은 브라우저 localStorage에 평문으로 저장하지 않는다. 서버 사이드 설정 파일 또는 OS keychain과 같은 보안 저장소를 사용해야 한다.

---

### 4.6 실행 기록 (Feature Area: Execution History)

**FR-EH-1:** 프로젝트별 실행 기록이 리스트로 저장된다.  
**FR-EH-2:** 각 기록 항목에는 요청 내용 요약, 실행 시작 시간, 실행 소요 시간, 실행 상태(성공/실패), 주요 로그 요약이 포함된다.  
**FR-EH-3:** 기록을 클릭하면 상세 화면에서 전체 요청, 응답, 로그, 변경 파일 목록, 승인/거절 내역을 확인할 수 있다.  
**FR-EH-4:** 실패한 실행 기록에서 원인 로그를 확인하고 동일 요청을 재실행할 수 있다.  
**FR-EH-5:** 각 기록에 승인/거절 내역이 별도로 기록된다.

---

### 4.7 승인 확인 (Feature Area: Approval Flow)

**FR-AF-1:** 파일 수정, 패키지 설치, 외부 명령 실행처럼 사용자 확인이 필요한 작업이 발생하면 CLI 실행이 일시 중지되고 채팅 화면 내에 승인 카드가 표시된다.  
**FR-AF-2:** 승인 카드에는 실행 예정 작업 내용, 대상 파일 또는 명령어, 위험도 수준, 설명이 명확히 표시된다.  
**FR-AF-3:** 승인 버튼과 거절 버튼이 승인 카드 내에 제공된다.  
**FR-AF-4:** 사용자가 승인하면 CLI 실행이 계속 진행되고, 거절하면 해당 작업이 중단된다.  
**FR-AF-5:** 승인 또는 거절 결과가 채팅 흐름과 실행 기록에 남는다.  
**FR-AF-6 (Resilience Constraint):** 승인 대기 중 사용자가 페이지를 새로 고치거나 연결이 끊기면, 재접속 시 미결 승인 요청이 복원되어 표시되어야 한다. (구현 전략은 아키텍처 단계에서 결정한다.)

---

### 4.8 온보딩 (Feature Area: Onboarding)

**FR-OB-1:** 처음 접속한 사용자(필수 설정 미완료 상태)는 4단계 설정 마법사를 거친다.  
**FR-OB-2:** 마법사 단계: ① CLI 도구 경로 설정 → ② API 키 또는 인증 정보 입력 → ③ 기본 프로젝트 폴더 선택 → ④ 설정 테스트.  
**FR-OB-3:** 설정 테스트 성공 후 프로젝트 선택 화면으로 자동 이동한다.  
**FR-OB-4:** 설정 테스트 실패 시 실패 원인과 수정 방법을 안내한다.

## 5. Screen Inventory and Navigation

| 화면 | 설명 | 진입 조건 |
|------|------|-----------|
| **온보딩 마법사** | 4단계 필수 설정 마법사 | 최초 접속 또는 필수 설정 미완료 |
| **프로젝트 선택** | 등록된 프로젝트 목록 (chat-first inbox 레이아웃) | 설정 완료 후 기본 화면 |
| **메인 채팅/실행** | 좌측 사이드바 + 중앙 채팅 + 우측 로그 패널 | 프로젝트 선택 시 |
| **실행 로그 패널** | 우측에 필요 시에만 열리는 패널 | 채팅 화면 내 on-demand |
| **승인 확인 UI** | 채팅 흐름 내 인라인 승인 카드 | 민감 작업 발생 시 |
| **실행 기록** | 프로젝트별 기록 리스트 + 상세 화면 | 좌측 사이드바 메뉴 |
| **설정** | CLI 경로, API 키, 모델, 실행 옵션 설정 | 좌측 사이드바 메뉴 또는 온보딩 |

**UX Constraint:** 모든 화면에서 사용자는 "프로젝트 선택 → 요청 입력 → 실행 결과 확인" 흐름만으로 핵심 기능을 사용할 수 있어야 한다. 로그, 파일 변경, 명령어는 기본 화면을 복잡하게 만들지 않는 방식으로 노출한다.

**Design Differentiation:** 프로젝트 선택 화면은 "chat-first inbox" 아키타입을 따른다. 단순 카드 나열이 아니라, 각 프로젝트 항목이 메시지 스레드처럼 보이며 최근 실행 요약과 상태가 인박스 스타일로 표시되어야 한다. 일반 대시보드 레이아웃과 레이아웃 실루엣이 명확히 달라야 한다.

## 6. Acceptance / Success Criteria Mapping

### AC-PM: 프로젝트 등록 및 선택

- [ ] 웹 UI에서 로컬 폴더 경로를 입력하거나 탐색하여 프로젝트를 등록할 수 있다.
- [ ] 등록된 프로젝트 목록이 표시되며 각 항목에 이름, 경로, 최근 실행 시간, 최근 상태가 보인다.
- [ ] "새 프로젝트 추가" 버튼이 항상 표시된다.
- [ ] 프로젝트 클릭 시 메인 채팅/실행 화면으로 이동한다.
- [ ] 프로젝트 A의 실행 기록이 프로젝트 B에 혼재되지 않는다.
- [ ] 빈 상태에서 첫 프로젝트 등록 안내 메시지가 표시된다.

### AC-EE: CLI 실행 Wrapper

- [ ] 채팅창 입력이 CLI 명령으로 변환되어 실행된다.
- [ ] 실행 상태 표시가 대기 중 / 실행 중 / 완료 / 실패 중 하나로 항상 표시된다.
- [ ] 실행 중 "중단" 버튼이 표시되고, 클릭 시 CLI 프로세스가 종료된다.
- [ ] 실패 시 채팅 화면에 에러 메시지와 관련 로그가 표시된다.
- [ ] 사용자 입력이 쉘 명령에 직접 연결되지 않는다 (파라미터화 검증).

### AC-LO: 실시간 로그

- [ ] 실행 중 CLI 출력이 지연 없이 UI에 스트리밍된다.
- [ ] 파일 변경, 명령 실행, 오류가 시각적으로 구분된다.
- [ ] 긴 로그가 기본 요약으로 표시되고 펼치기로 전체를 볼 수 있다.
- [ ] 오류 로그에 구분 스타일이 적용된다.

### AC-CI: 채팅형 작업 요청

- [ ] 자연어 입력이 가능하고 입력창이 화면 하단에 고정된다.
- [ ] 사용자 요청과 에이전트 응답이 대화 흐름으로 표시·저장된다.
- [ ] 이전 대화 기록을 스크롤하여 확인할 수 있다.
- [ ] 이전 요청을 재실행하거나 수정 후 재실행할 수 있다.

### AC-CF: 설정 관리

- [ ] CLI 경로, API 키, 기본 모델, 기본 폴더, 실행 옵션을 저장할 수 있다.
- [ ] 앱 재시작 후에도 설정값이 유지된다.
- [ ] 필수 설정 누락 시 실행 전에 어떤 설정이 필요한지 안내한다.
- [ ] "설정 테스트" 버튼이 성공/실패 결과를 사용자 친화적 메시지로 표시한다.
- [ ] API 키가 브라우저 localStorage 평문으로 저장되지 않는다.

### AC-EH: 실행 기록

- [ ] 프로젝트별 실행 기록 리스트가 저장된다.
- [ ] 각 기록 항목에 요청 요약, 시간, 상태, 소요 시간이 표시된다.
- [ ] 기록 클릭 시 전체 요청, 응답, 로그, 변경 파일, 승인 내역이 보인다.
- [ ] 실패 기록에서 원인 확인 후 재실행 버튼이 제공된다.

### AC-AF: 승인 확인

- [ ] 민감 작업 발생 시 CLI 실행이 멈추고 채팅 내 승인 카드가 표시된다.
- [ ] 승인 카드에 작업 내용, 대상, 위험도, 설명이 표시된다.
- [ ] 승인 클릭 시 실행이 계속된다.
- [ ] 거절 클릭 시 해당 작업이 중단된다.
- [ ] 승인/거절 내역이 실행 기록에 저장된다.
- [ ] 승인 대기 중 페이지 새로 고침 후 미결 승인 요청이 복원된다.

### AC-OB: 온보딩

- [ ] 최초 접속 시 4단계 마법사가 표시된다.
- [ ] 각 단계가 순차적으로 안내된다.
- [ ] 테스트 성공 시 프로젝트 선택 화면으로 이동한다.
- [ ] 테스트 실패 시 원인과 수정 방법이 표시된다.

## 7. UX & Design Requirements

### 7.1 UX 원칙 (Non-negotiable)

- CLI를 모르는 사용자가 버튼, 입력창, 상태 표시만으로 모든 핵심 기능을 사용할 수 있어야 한다.
- 로그, 명령어, 파일 변경 내역은 기본 화면을 복잡하게 만들지 않는 방식으로 제공한다 (on-demand 노출).
- 실행 상태, 실패 원인, 승인 필요 여부가 사용자가 즉시 이해할 수 있는 언어와 시각으로 표시된다.

### 7.2 Copy Strategy

- 버튼 레이블은 "Submit" 같은 일반 문구 대신 "작업 실행", "다시 실행", "변경 승인" 처럼 동작을 명확히 설명한다.
- 빈 상태 메시지는 "데이터가 없습니다" 대신 "첫 번째 프로젝트를 추가해 에이전트를 실행해 보세요" 처럼 다음 행동을 안내한다.
- 오류 메시지는 에러 코드 대신 문제 내용과 다음 안전 행동을 설명한다.

### 7.3 State Contract

| State | 표시 요건 |
|-------|----------|
| empty | 사용자가 무엇을 먼저 해야 하는지 안내 |
| loading | 진행 중임을 표시하고 기본 흐름이 유지되어야 함 |
| error | 문제 설명 + 다음 안전 행동 제시 |
| success | 완료된 내용 확인 + 변경된 상태 표시 |
| completed | 최종 결과 또는 핸드오프 상태를 명확히 표시 |

## 8. Architecture Constraints (PRD Level)

이 섹션은 아키텍처 결정을 내리지 않는다. UX 요구사항에서 파생된 기술적 제약을 아키텍처 단계에 전달하는 목적이다.

| 제약 | 출처 |
|------|------|
| 실시간 로그 스트리밍을 위한 WebSocket 또는 SSE 필요 | FR-LO-1 |
| 서버 사이드 CLI 프로세스 실행·모니터링·종료 능력 필요 | FR-EE-1, FR-EE-3 |
| 서버가 로컬 파일시스템 경로를 읽을 수 있어야 함 (브라우저 직접 접근 불가) | FR-PM-1 |
| API 키 등 민감 설정의 보안 저장소 필요 | FR-CF-6 |
| 승인 대기 중 연결 끊김 시 상태 복원 메커니즘 필요 | FR-AF-6 |
| CLI 명령 구성 시 Command Injection 방지 필수 | FR-EE-5 |

## 9. Risks and Open Questions

### 9.1 Critical Risks

| # | Risk | Impact | Mitigation Direction |
|---|------|--------|---------------------|
| R1 | **Command Injection:** 사용자 입력이 CLI 명령에 직접 삽입될 경우 임의 명령 실행 가능 | 높음 (보안) | 명령 인수 파라미터화 필수; 아키텍처 단계에서 구현 패턴 지정 |
| R2 | **API 키 노출:** 민감 설정값이 브라우저에 평문 저장되거나 로그에 출력될 경우 | 높음 (보안) | 서버 사이드 저장, 로그 필터링 요건을 아키텍처 단계에서 확정 |
| R3 | **승인 대기 중 연결 끊김:** 사용자 재접속 시 미결 승인 요청 손실 또는 CLI 프로세스 무한 대기 | 중간 (안정성) | FR-AF-6 복원 요건 포함; 구현 전략은 아키텍처 단계에서 결정 |
| R4 | **장시간 실행 프로세스:** 수 분 이상 실행 시 연결 유지, 타임아웃, 중간 상태 복원 | 중간 (안정성) | 타임아웃 정책 및 상태 복원 전략을 아키텍처 단계에서 정의 |

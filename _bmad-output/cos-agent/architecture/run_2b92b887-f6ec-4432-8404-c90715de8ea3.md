## 1. 핵심 아키텍처 결정 (OQ2 해소)

### 결정: 로컬 서버 + 웹 브라우저

**선택 근거:**

| 옵션 | 장점 | 단점 |
|------|------|------|
| **로컬 서버 + 브라우저** (선택) | 설치 단순, 개발/테스트 빠름, 업데이트 용이, CLI 도구와 동일한 환경 | 서버 시작 명령 필요, 파일 탐색 UX 제한 |
| Electron 앱 | 파일 탐색 네이티브 UX, 앱처럼 실행 | 빌드 복잡, 배포 무거움, 초기 개발 비용 높음 |

**결정 내용:** Node.js 백엔드 서버가 `localhost:3000`에서 실행되며, React SPA를 서빙한다. 사용자는 `npx agent-studio` 또는 `npm start` 명령으로 서버를 시작하고 브라우저에서 접속한다. 파일 탐색은 서버 사이드 API를 통해 디렉토리 목록을 제공하는 방식으로 구현한다.

### 3.3 Process Manager (핵심 컴포넌트)

CLI 프로세스의 라이프사이클을 관리한다. **이 컴포넌트가 시스템의 핵심이다.**

```
ProcessManager
├── spawn(command, args[], cwd)         → child_process.spawn (shell: false 필수)
├── kill(executionId)                  → SIGTERM → 타임아웃 후 SIGKILL
├── write(executionId, input)          → stdin 주입 (승인 응답)
├── getStatus(executionId)             → 현재 상태 반환
└── on('data', handler)                → stdout/stderr 이벤트 → WebSocket 브로드캐스트
```

**Command Injection 방지 (FR-EE-5):**
```
// 금지
exec(`openclaw "${userInput}"`)

// 필수
spawn('openclaw', ['--prompt', userInput, '--project', projectPath], { shell: false })
```

모든 CLI 명령은 `child_process.spawn`의 인수 배열로만 구성한다. `shell: true` 옵션은 절대 사용하지 않는다.

### 3.4 Settings Manager

| 설정 유형 | 저장 위치 | 이유 |
|----------|----------|------|
| API 키 | OS keychain (`keytar`) | FR-CF-6: localStorage/평문 파일 저장 금지 |
| CLI 실행 경로 | `~/.agent-studio/config.json` | 민감하지 않은 일반 설정 |
| 기본 모델, 실행 옵션 | `~/.agent-studio/config.json` | 동일 |
| 기본 프로젝트 폴더 | `~/.agent-studio/config.json` | 동일 |

keytar 사용이 불가한 환경(CI, Docker 등)을 위한 fallback으로 환경변수(`AGENT_STUDIO_API_KEY`) 지원을 제공한다.

## 4. 데이터 모델 (SQLite)

데이터베이스 위치: `~/.agent-studio/db.sqlite`

### 4.1 테이블 스키마

```sql
-- 프로젝트 테이블
CREATE TABLE projects (
    id          TEXT PRIMARY KEY,  -- uuid
    name        TEXT NOT NULL,
    path        TEXT NOT NULL UNIQUE,
    created_at  TEXT NOT NULL,     -- ISO 8601
    last_run_at TEXT,
    last_status TEXT               -- 'completed' | 'failed' | 'cancelled' | null
);

-- 실행 테이블
CREATE TABLE executions (
    id            TEXT PRIMARY KEY,
    project_id    TEXT NOT NULL REFERENCES projects(id),
    request_text  TEXT NOT NULL,
    status        TEXT NOT NULL,   -- 'pending' | 'running' | 'approval_pending' | 'completed' | 'failed' | 'cancelled'
    started_at    TEXT,
    completed_at  TEXT,
    duration_ms   INTEGER,
    summary       TEXT,
    error_message TEXT
);

-- 메시지 테이블 (채팅 흐름)
CREATE TABLE messages (
    id           TEXT PRIMARY KEY,
    execution_id TEXT NOT NULL REFERENCES executions(id),
    project_id   TEXT NOT NULL REFERENCES projects(id),
    type         TEXT NOT NULL,    -- 'user' | 'agent' | 'system' | 'approval_request' | 'approval_result'
    content      TEXT NOT NULL,
    metadata     TEXT,             -- JSON (파일 변경 목록 등)
    created_at   TEXT NOT NULL
);

-- 로그 테이블
CREATE TABLE logs (
    id           TEXT PRIMARY KEY,
    execution_id TEXT NOT NULL REFERENCES executions(id),
    timestamp    TEXT NOT NULL,
    level        TEXT NOT NULL,    -- 'info' | 'warn' | 'error'
    category     TEXT NOT NULL,    -- 'stdout' | 'stderr' | 'file_change' | 'command'
    content      TEXT NOT NULL
);

-- 승인 이벤트 테이블
CREATE TABLE approvals (
    id            TEXT PRIMARY KEY,
    execution_id  TEXT NOT NULL REFERENCES executions(id),
    action_type   TEXT NOT NULL,   -- 'file_modify' | 'package_install' | 'command_exec'
    target        TEXT NOT NULL,   -- 파일 경로 또는 명령어
    risk_level    TEXT NOT NULL,   -- 'low' | 'medium' | 'high'
    description   TEXT NOT NULL,
    status        TEXT NOT NULL,   -- 'pending' | 'approved' | 'rejected'
    requested_at  TEXT NOT NULL,
    decided_at    TEXT
);
```

### 4.2 실행 기록 보존 정책 (OQ5 해소)

- 기본값: 프로젝트당 최근 100개 실행 기록 보존
- 로그 테이블: 실행당 최대 10,000줄 (초과분은 요약으로 대체)
- 사용자 설정으로 보존 한도 조정 가능 (설정 화면에 선택적 노출)

## 5. 실시간 통신 설계 (WebSocket)

### 5.1 연결 관리

- 클라이언트는 페이지 로드 시 단일 WebSocket 연결 수립
- 연결 끊김 시 지수 백오프(1s → 2s → 4s → ... → 최대 30s)로 자동 재연결
- 재연결 성공 시 현재 활성 실행이 있으면 해당 실행의 스트림에 자동 재구독

### 5.2 메시지 타입

```typescript
// 서버 → 클라이언트
type ServerMessage =
  | { type: 'log'; executionId: string; data: LogEntry }
  | { type: 'status'; executionId: string; status: ExecutionStatus }
  | { type: 'approval_request'; executionId: string; approval: ApprovalRequest }
  | { type: 'execution_complete'; executionId: string; summary: string }
  | { type: 'connected'; pendingApprovals: ApprovalRequest[] }  // 재연결 시 미결 복원

// 클라이언트 → 서버
type ClientMessage =
  | { type: 'subscribe'; executionId: string }
  | { type: 'unsubscribe'; executionId: string }
```

### 5.3 승인 대기 복원 (FR-AF-6)

재연결 시 서버는 DB에서 `status = 'pending'`인 `approvals` 레코드를 조회하고 `connected` 메시지에 포함하여 전송한다. 클라이언트는 이를 채팅 흐름에 ApprovalCard로 복원한다.

## 6. 실행 상태 머신

```
                   ┌──────────┐
                   │ PENDING  │
                   └────┬─────┘
                        │ 실행 시작
                   ┌────▼─────┐
              ┌────│ RUNNING  │────┐
              │    └────┬─────┘    │
              │         │ 승인 필요 │ 중단 버튼
              │    ┌────▼──────┐   │
              │    │ APPROVAL  │   │
              │    │ _PENDING  │   │
              │    └────┬──────┘   │
              │    승인  │  거절    │
              │    ┌────▼──┐  ┌───▼──────┐
              │    │(계속) │  │CANCELLED │
              │    └───────┘  └──────────┘
              │ 성공          │ 실패
         ┌────▼──────┐  ┌────▼─────┐
         │ COMPLETED │  │  FAILED  │
         └───────────┘  └──────────┘
```

상태 전이는 서버에서만 권한을 갖는다. 클라이언트는 상태를 변경할 수 없고 서버의 `status` 이벤트를 구독한다.

## 8. 보안 아키텍처

### 8.1 Command Injection 방지 (FR-EE-5, R1)

- `child_process.spawn`만 사용, `exec`/`execSync` 사용 금지
- `shell: false` 옵션 강제
- 사용자 입력은 항상 인수 배열의 개별 요소로 전달
- CLI 명령 구성 로직을 별도 `CommandBuilder` 모듈로 분리하여 단위 테스트 작성

### 8.2 API 키 보안 저장 (FR-CF-6, R2)

- API 키는 `keytar`를 통해 OS 키체인에 저장 (macOS Keychain, Linux Secret Service, Windows Credential Manager)
- 설정 조회 API에서 API 키는 `"*****"` 마스킹 형태로만 반환
- CLI 실행 시 API 키는 환경변수로 자식 프로세스에 주입 (명령줄 인수 금지 — `ps` 명령으로 노출될 수 있음)
- 로그 필터링: stdout/stderr에서 API 키 패턴을 정규식으로 탐지·마스킹 후 DB 저장 및 WebSocket 전송

### 8.3 파일시스템 접근 제한

- 파일시스템 탐색 API는 사용자가 등록한 프로젝트 경로와 홈 디렉토리 내부만 허용
- 경로 순회 공격(`../../etc/passwd` 등) 방지: `path.resolve`로 정규화 후 허용 경로 접두사 확인

## 9. 승인 흐름 상세 설계

### 9.1 승인 트리거 방식 (OQ4 해소)

초기 버전은 두 가지 방식을 병렬 지원한다:

1. **CLI 프로토콜 기반 (우선):** CLI 도구(예: OpenClaw)가 stdout에 구조화된 승인 요청 메시지를 출력하고, 승인 응답을 stdin으로 받는 방식. 이 방식은 CLI 도구의 --print 플래그 등 JSON 출력 모드를 활용한다.

2. **패턴 감지 기반 (폴백):** stdout에서 `"Do you want to proceed?"`, `"Press y to confirm"` 같은 패턴을 감지하면 CLI 프로세스를 일시 정지하고 승인 UI를 표시한다. 사용자 응답을 stdin으로 주입한다.

**구현 우선순위:** 1번 방식으로 설계하되, 지원 CLI 도구 확정 후(OQ1) 패턴 사전을 추가한다.

### 9.2 승인 흐름 시퀀스

```
CLI Process        Process Manager       WebSocket          Frontend
    │                    │                   │                  │
    │ stdout: approval   │                   │                  │
    ├───────────────────►│                   │                  │
    │                    │ pause process     │                  │
    │                    │ save to DB        │                  │
    │                    │──────────────────►│ approval_request │
    │                    │                   ├─────────────────►│
    │                    │                   │                  │ show ApprovalCard
    │                    │                   │                  │
    │                    │                   │◄─ REST POST /api/approvals/:id/respond
    │                    │◄── update DB ─────┤                  │
    │ stdin: "y"/"n"     │                   │                  │
    │◄───────────────────┤                   │                  │
    │                    │──────────────────►│ status: running  │
    │                    │                   ├─────────────────►│
```

### 10.2 온보딩 판정 로직

앱 초기화 시 다음 조건을 순서대로 확인한다:

1. `config.json` 파일 존재 여부
2. `cli_path` 필드가 설정되어 있고 해당 경로의 바이너리가 실행 가능한지
3. API 키가 keychain에 등록되어 있는지

하나라도 실패하면 온보딩 마법사로 리다이렉트한다.

**필수 설정 항목 (OQ3 해소):** CLI 실행 경로 + API 키. 기타 설정(기본 모델, 기본 프로젝트 폴더, 실행 옵션)은 선택 항목으로 설정 화면에서 나중에 변경 가능하다.

## 12. 지원 CLI 도구 범위 (OQ1 임시 결정)

초기 릴리스는 **OpenClaw (Claude Code)** 단일 도구를 대상으로 구현한다. CLI 도구별 출력 파싱 로직을 `CliAdapter` 인터페이스로 추상화하여 추후 다른 도구를 추가할 수 있게 한다.

```typescript
interface CliAdapter {
  buildArgs(request: string, options: ExecutionOptions): string[];
  parseApprovalRequest(line: string): ApprovalRequest | null;
  parseResult(lines: string[]): ExecutionSummary;
}
```

OpenClaw의 JSON 출력 모드(`--output-format stream-json` 또는 동등한 플래그)를 기본으로 사용한다. JSON 출력이 불가한 경우 텍스트 패턴 매칭 폴백을 사용한다.

## 13. 구현 위험 및 트레이드오프

### 13.1 장시간 실행 프로세스 (R4)

| 위험 | 영향 | 완화 전략 |
|------|------|----------|
| 브라우저 탭 닫힘으로 WebSocket 끊김 | 실행은 계속되지만 UI에서 결과 수신 불가 | 서버는 실행을 독립적으로 관리; 재연결 시 missed 로그를 DB에서 재전송 |
| 수 분 이상 실행 시 HTTP 타임아웃 | 실행 시작 API 응답 지연 | 실행 시작 API는 즉시 executionId를 반환하고 비동기로 진행; WebSocket으로 상태 추적 |
| 프로세스 누수 (서버 재시작 시) | 고아 CLI 프로세스 발생 | 서버 시작 시 DB의 `status = 'running'` 실행을 `failed`로 마킹; PID 기반 프로세스 정리 |

### 13.2 로그 스트리밍 성능

- 초당 수백 줄의 로그가 발생할 수 있음 (특히 에러 케이스)
- 완화: 서버에서 16ms 단위로 로그를 배치 후 단일 WebSocket 메시지로 전송 (디바운스)
- 클라이언트에서는 가상 스크롤링(`react-virtual`)으로 대량 로그 렌더링 처리

### 13.3 승인 패턴 감지 신뢰도

- 패턴 기반 승인 감지는 false positive 위험이 있음
- 완화: 화이트리스트 패턴만 사용; 의심스러운 라인은 일반 로그로 처리하고 사용자가 수동으로 진행 여부를 판단
- CLI 프로토콜 기반 방식이 확정되면 패턴 감지는 레거시 폴백으로 격하

### 13.4 OS keychain 의존성

- keytar는 OS keychain 데몬이 필요하며, 일부 최소 Linux 환경에서 누락될 수 있음
- 완화: keytar 초기화 실패 시 설정 화면에 명확한 안내와 함께 파일 기반 암호화 저장소 fallback 제공 (적절히 경고 표시)
